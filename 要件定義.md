## プロジェクト概要

- **プロジェクト名**: VeggiePapa
- **コンセプトコピー**: 花束の代わりに大根を。パパって何野菜？
- **コア体験**:  
  チャット形式で「お父さんのこと」をAIと一緒に振り返り、そこから生まれた「野菜のキャラクター」とレシピを通じて、父の日に「一緒に食べる」行動を優しく後押しする。

---

## 0. 目的・ゴール（プロトタイプ版）

- **目的（体験のゴール）**
  - 子どもが、お父さんの性格や思い出、健康への心配をあらためて言語化し、「お父さんを観察する」きっかけをつくる。
  - その結果として、「お父さんを野菜に例えた診断」と「一緒に食べるための招待状」を生み出し、父の日に連絡や食事のきっかけをつくる。

- **今回のプロトタイプで実現すること**
  - Webブラウザ上で、以下の一連のフローを **ストレスなく体験できる** ことをゴールとする。
    1. サービスの趣旨を直感的に理解する（タイトル〜導入）
    2. AIとの対話で「父の断片」を入力する（対話型プロファイリング）
    3. 「お父さん＝○○野菜」診断と、その理由テキスト＋**絵本風イラスト（画像生成）**を受け取る（野菜決定）
    4. その野菜を使った簡単レシピと、「お父さんを誘うメッセージ案」を受け取る（招待状）
  - **画像生成（野菜コスプレのお父さん＋子ども）は本プロトタイプで必須とし、アニメーションは簡易版でもよい** とする。

---

## 1. 対象ユーザー・利用シーン

- **対象ユーザー**
  - 20〜39歳の子ども世代
  - 50〜60代の父親を持つ人
  - 親と離れて暮らしているケースが多い
  - 普段は「用事があるときだけ連絡する」関係
  - 感謝や心配はあるが、直接伝えるのは照れくさい

- **利用シーン（想定）**
  - 父の日の数日前〜当日、SNSや友人経由で「うちの父、大根だったｗ」などの投稿を見てアクセスする。
  - 夜、自宅でスマホもしくはPCから1人で体験する。
  - 体験後、「診断結果＋メッセージ案＋レシピ」をスクショや共有リンクで父に送る。

## 2. ユーザーフロー（画面単位）

### 2-1. Scene 1：タイトル／導入ページ

- **目的**
  - 「何をするサービスか」を一瞬で伝え、開始ハードルを下げる。

- **UI・コピー（最小構成）**
  - 画面中央にシンプルな植木鉢イラスト（絵本風）。
  - 見出しテキスト：
    - 「お父さんを、野菜に例えてみない？」
    - サブコピー：  
      「お父さんに似た“野菜”を育てて、一緒に食べる父の日の招待状をつくるアプリです。」
  - ボタン：
    - 「はじめる」ボタン（コールトゥアクション）

- **機能要件**
  - 「はじめる」クリックで Scene 2（対話画面）へ遷移。
  - 遷移はフェードやスライドなど、簡単なアニメーションで「ページをめくる」感覚を演出（Framer Motion想定）。

---

### 2-2. Scene 2：対話型プロファイリング（AIチャット）

- **目的**
  - ユーザーが「お父さんの断片」を自然に思い出して入力できるようにする。

- **UI**
  - 画面中央：小さな芽〜茎のイラスト（成長アニメーションは簡易でも可）。
  - 画面下部：チャットUI風の質問＋テキスト入力欄。
  - ヘッダー近く：マスコットキャラ（野菜妖精「ベジまる」）と短いガイドテキスト。

- **質問フロー（最低限必要なもの）**
    1. あなたのお父さんを一言で表すと？（例：頑固、優しい、マイペース、職人気質）
    2. 子どもの頃、お父さんと一緒にしたことで覚えていることは？（例：絵本を読んでもらった）
    3. お父さんのちょっと困ったところ、でも嫌いになれないところは？（例：怒りっぽい、愛情表現が少ない）
    4. お父さんに大事にされてると感じた瞬間は？（例：迎えに来てくれた）
    5. これからも、お父さんにどんな人でいてほしい？（例：元気でいてほしい）

- **インタラクション（最小版）**
  - 1質問ごとに、芽→茎→葉のようにイラストが少し成長する。
  - 成長状態はフロントエンド側の状態管理のみでOK（AI連動でなくてもよい）。

- **機能要件**
  - 各質問に対するユーザーの回答をフロント側で配列として保持。
  - ユーザーが質問への回答を入力し、エンターを押したら、次の質問へ。
  - 全質問完了時に、まとめたテキストをLLM APIへ送信し、野菜診断のためのレスポンスを取得する。

---

### 2-3. Scene 3：野菜診断結果（テキスト＋画像生成必須）

- **目的**
  - 「あなたのお父さんは○○野菜」という分かりやすいメタファーで、父の存在を新しく捉え直してもらう。

- **表示内容（必須）**
  - キャッチコピー例：
    - 「あなたのお父さんは、『どっしり大根パパ』。」
  - 説明テキスト：
    - LLMからの出力をもとに、「無口だけど家族を支える芯の強さが、大根みたいです。」などの理由文。
  - 画像
    - 対話内容と画像生成APIを用いて「野菜コスプレのお父さん＋子ども」の絵本風イラストを生成し表示する。    
    - 生成に時間がかかる場合は、ローディング演出（簡易アニメーションやテキスト）で待ち時間を演出する。

- **機能要件**
  - LLMのレスポンス構造（例）：
    - `vegetable_type`: "大根"
    - `nickname`: "どっしり大根パパ"
    - `traits`: ["どっしり", "不器用だけど優しい"]
    - `reason_text`: 説明文
    - `father_inner_voice`: お父さんの本音テキスト
    - `invite_message`: 子どもから父への招待メッセージ案
    - `recipe_title`, `recipe_body`: レシピ情報
  - 上記のうち、このSceneでは `vegetable_type` / `nickname` / `traits` / `reason_text` を表示。
  - 「つぎへ（お父さんの本音を見る）」ボタンで Scene 4 へ。

---

### 2-4. Scene 4：父の本音テキスト表示（簡易版）

- **目的**
  - 「野菜」というメタファーを通じて、直接は聞けない父の本音を想像する瞬間をつくる。

- **表示内容**
  - 見出し例：
    - 「君のお父さん、もしかしたら...」
  - 本文：
    - LLMからの `father_inner_voice` を、手書き風フォント風のスタイルで表示。
    - 例：「本当はもっと一緒にご飯を食べたいけど、どう誘っていいか分からないのかもね。」

- **機能要件**
  - テキストのタイピング風アニメーション（シンプルな一文字ずつ表示）を入れられるとベター。
  - 「一緒に食べる招待状をつくる」ボタンで Scene 5 へ。

---

### 2-5. Scene 5：レシピ＆招待状画面（共有前の最終アウトプット）

- **目的**
  - 体験を「父との行動」に接続するための、具体的な一歩（メッセージ＋レシピ）をつくる。

- **表示内容（必須）**
  - 野菜タイプとニックネーム（再掲）：
    - 例：「今年のあなたのお父さんは、『どっしり大根パパ』でした。」
  - レシピ情報：
    - `recipe_title`（例：豚バラ大根）
    - `recipe_body`（材料・簡易手順・健康面のひとこと）
  - 招待メッセージ案：
    - `invite_message` 例：  
      「どっしり大根みたいなパパが大好きです。飲みすぎないで長生きしてほしいから、今度この『豚バラ大根』一緒に食べない？」
  - 共有用UI：
    - メッセージ案をコピーできる「テキストをコピー」ボタン。
    - （余力があれば）結果画面のURLシェアボタン or 画像化ボタン。

- **機能要件**
  - 「テキストをコピー」クリック時に、診断結果＋メッセージをクリップボードへコピー。
  - 画面上に「コピーしました」のトースト表示。

## 3. 機能要件（バックエンド／AI連携の最小要件）

- **LLM連携**
  - Gemini API でLLMテキスト生成, 画像生成を行う。
    - テキスト: Gemini 1.5 Pro（会話→野菜診断→レシピ・本音テキストまで生成）
    - 画像: Gemini 画像モデル（Nano Banana 2）
  - インプット：
    - ユーザー回答一覧（質問文＋回答文）
  - アウトプット（JSON想定）：
    - `vegetable_type`: 例）"大根", "キャベツ", "ブロッコリー" など
    - `nickname`: 例）"どっしり大根パパ"
    - `traits`: 2〜4個のキーワード配列
    - `reason_text`: 野菜と父の関係を説明するテキスト
    - `father_inner_voice`: 父の本音を推論したテキスト
    - `recipe_title`: レシピ名
    - `recipe_body`: 材料・手順・健康ひとことを含むテキスト
    - `invite_message`: 子どもから父に送る招待メッセージ案

- **画像生成（必須／スコープ調整用）**
  - 可能であれば、野菜タイプとユーザーの父の特徴をもとに、画像生成APIを一度叩く。

## 4. 技術スタック（プロトタイプ）

- **フロントエンド**
  - Next.js / React（SPA的な滑らかなページ遷移）
  - TypeScript（任意）

- **スタイリング・アニメーション**
  - ベースは通常のCSS（または好みでTailwind CSS）でOK。
  - アニメーションはCSSアニメーションで最低限の演出（フェードイン／サイズ変化など）を実装。
  - 余裕があれば、Lottie（lottie-react等）で絵本風イラストを動かすことも検討（必須ではない）。

- **バックエンド / API連携**
  - Next.js API Routes を使用し、サーバー側からLLM APIを呼び出す構成。
  - .env にAPIキーを保存し、フロントに直接露出させない。

## 7. ディレクトリ構成（`app` 配下の最小構成）

Next.js の `app` ルーターを前提にした、VeggiePapa プロト向けの最小構成。

```text
app/
  layout.tsx            // 全ページ共通レイアウト（フォント、背景、ヘッダーなど）
  globals.css           // 全体のスタイル（絵本っぽい色・フォント指定など）

  page.tsx              // Scene 1：タイトル／導入ページ

  flow/
    page.tsx            // Scene 2〜5 をまとめて扱うウィザード画面のコンテナ

  assets/
    images/             // UIで使う静的画像（ロゴ、背景テクスチャ、固定イラストなど）
    lottie/             // Lottie JSON（後から追加する場合）

  api/
    diagnose/route.ts   // POST /api/diagnose
                         // ユーザー回答を受け取り、Geminiで
                         // ・野菜診断
                         // ・父の本音
                         // ・レシピ
                         // ・招待メッセージ
                         // ・画像用プロンプト
                         // をJSONで返す

components/
  SeedIntro.tsx         // Scene 1 用の「植木鉢＋ベジまる」などの表示コンポーネント（任意）
  FlowWizard.tsx        // Scene 2〜5 全体を制御するコンポーネント
  QuestionStep.tsx      // Scene 2：質問と回答入力コンポーネント
  ResultVeggieCard.tsx  // Scene 3：野菜診断＋イラスト表示コンポーネント
  InnerVoiceView.tsx    // Scene 4：父の本音テキスト表示コンポーネント
  InvitationView.tsx    // Scene 5：レシピ＋招待状表示コンポーネント
  LoadingOverlay.tsx    // LLM/画像生成待ちのローディング表示

lib/
  gemini.ts             // Gemini API クライアントラッパー（テキスト＆画像）
  prompts.ts            // LLM に渡すプロンプトテンプレートと型定義

types/
  diagnosis.ts          // LLMのJSONレスポンス型
```

- 画像やLottieなどの**フロント側のアセット**は、`app/assets/images` / `app/assets/lottie` もしくは `public/veggiepapa/...` のどちらかに置く。  
  - 「Next.js 標準の静的配信」に寄せたい場合は `public/veggiepapa/images/...` を使う。
  - `app/assets/...` に置く場合は、`import` して使うReactコンポーネント内アセットとして扱う。

- **シーン遷移の考え方**
  - `app/page.tsx` … 入口（Scene 1）。「はじめる」ボタンで `/flow` へ遷移。
  - `app/flow/page.tsx` … Scene 2〜5 をステップ管理（ローカルstateで `step` を持つ）。
  - Scene 2 の回答が揃ったタイミングで `/api/diagnose` を叩き、返ってきたJSONをもとに Scene 3〜5 で表示。

## 5. 非機能要件（プロト版の前提）

- **対応デバイス**
  - スマホ（縦画面）を主ターゲット。ただしPCブラウザでも破綻しないレスポンシブ対応。

- **パフォーマンス**
  - LLMレスポンス待ち時間を考慮し、「生成中」インジケータ（ベジまるの一言＋ローディング）を表示。

- **トラッキング（任意）**
  - 時間があれば、どのSceneまで到達したかの簡易ロギング（コンソールや簡易なイベント送信）を用意。
